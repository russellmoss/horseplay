# derby-edge — model explanation & diagnostic gaps

This document answers the diagnostic questions you raised about the models
(calibration, SLAM_DUNK definition, exotic construction, stake sizing,
per-track calibration, drift interpretation, architecture).

**Honest framing up front.** Several of the questions ask for things we
don't currently produce — calibration plots, Brier/log-loss, CLV, ROI by
drift bucket, etc. Every one of those requires *historical settled data*
and an *evaluator script*, and neither exists today. derby-edge stores
live `RaceAnalysis` in an in-memory Map; the only persisted thing is
`data/locked-recs.json` (locked AI bet plans + settled outcomes). So
parts of this doc describe **what the model is**, parts describe **what
we'd need to build to actually answer the question**, and the gap is
called out explicitly each time.

Scope: this doc covers Win/Place/Show only. Exotics (exacta/trifecta) are
generated by the AI bet planner without explicit probability math — see
§C for the implications.

## A. Calibration

### A.1 — Show me the calibration plot per pWin decile, for pWin / pPlace / pShow

**Not currently produced.** We have no historical settled sample stored in
a queryable shape, and no script that buckets predicted probabilities and
compares them to realized frequencies. The data needed to build it
*does* exist on disk, though — every settled race in `data/locked-recs.json`
has the analysis snapshot at lock time AND the official outcome, so a
backfill is mechanical:

1. For every race in `data/locked-recs.json` where `settlement` exists:
   - For each active horse, extract `pWin`, `pPlace_harville`, `pPlace_henery`, `pShow_harville`, `pShow_henery` from the analysis at lock time.
   - Look up actual `finishPosition` from `race.results.runners`.
2. Bucket horses into 10 deciles by predicted probability.
3. In each bucket, compute realized win/place/show frequency.
4. Plot predicted (x) vs realized (y); a perfectly calibrated model lies on y=x.

**What we should expect.** Two known sources of miscalibration:

- **Favorite-longshot bias.** We apply a power correction (`p^1.06`) at
  the win-probability level. Literature alpha values cluster 1.05–1.10
  for North American thoroughbreds, but the *right* alpha for our
  scrape window (live odds at T-1:00, not closing odds) might differ.
  Worth fitting on real data.
- **Harville's known bias.** Harville assumes that after the winner is
  decided, the rest of the race is a fresh draw — which overstates
  favorites' place/show chances and understates longshots'. Henery
  corrects this with β=0.81, γ=0.65, but those are mid-literature
  defaults, not fit to our sample.

**Build cost:** ~half a day. One script in `scripts/` that reads
`locked-recs.json`, joins to analysis snapshots, emits a CSV. Plot in
the dashboard, or just pipe to a notebook. This is the highest-value
diagnostic we don't have.

### A.2 — Brier score and log-loss per model per pool, out-of-sample

**Not currently produced.** Same building blocks as A.1.

For a binary outcome `y_i ∈ {0, 1}` and predicted probability `p_i`:

```
Brier = (1/N) Σ (p_i − y_i)²
Log-loss = −(1/N) Σ [y_i log(p_i) + (1−y_i) log(1−p_i)]
```

Both numbers per model per pool (Harville-win, Harville-place,
Harville-show, Henery-place, Henery-show, plus the heuristic for
reference). Lower = better calibration. Useful because they collapse
the whole calibration curve into one number per model.

Caveats once we build it:

- **Out-of-sample matters.** The β/γ/α constants are hard-coded, not fit
  to our data, so today's metrics ARE effectively out-of-sample. If we
  ever fit β/γ to data, we must hold out a window.
- **Small N.** Per-day Brier/log-loss is noisy. Need rolling windows
  (last 30 days, last 200 races) to be meaningful.

### A.3 — CLV by model, by bet type

**Not currently produced** — and the data isn't even on disk. CLV
(closing-line value) requires capturing odds at the *moment we lock*
and again at the *closing instant* (post). We snapshot the analysis at
T-1:00 (lock time), but we do not currently re-snapshot at gates-open.
The poller switches to a 5-second cadence once `status === 'closed'`,
so the infrastructure is there — we just don't store the last pre-race
tick.

**Build cost:** small change to `lib/scraper/auto-settle.ts` /
`poller.ts` to persist the final pre-race `RaceAnalysis` alongside the
settlement. Then a CLV evaluator script.

CLV is the right metric for whether we're picking *value*, not whether
we're picking *winners*. Outcome luck on the 50-race sample you cited is
huge; CLV reveals true edge ~10× faster than realized ROI. Build this
before tweaking thresholds.

### A.4 — Harville vs Henery disagreement: who wins?

**Not currently produced as a standing report.** The settlement
infrastructure stores both `settlementByModel.harville` and
`settlementByModel.henery` per race (see `lib/store.ts`
`LockedRecommendation.settlementByModel`), so the join is trivial.

What the report should answer:

- Filter to races where Harville and Henery tagged DIFFERENT horses as
  the highest-edge or highest-probability pick.
- For each such race, did Harville's pick or Henery's pick finish higher?
- Aggregate: win rate of Harville's exclusive picks vs Henery's exclusive picks, when they disagree.

The math literature suggests Henery wins these on average for
non-favorites (the whole reason β<1 exists). Whether that holds on our
*scraped, FLB-corrected, T-1:00* input vector is an empirical
question we can answer with the data we already have.

**Build cost:** ~2 hours of script work.

## B. SLAM_DUNK threshold logic

### B.1 — Exact definition

From `lib/math/ev.ts`, `classifySignal()`:

```ts
if (placeEdge.harvilleFloor > 0 || showEdge.harvilleFloor > 0)
    → 'slam_dunk'
else if (placeEdge.harvilleMid > leanThreshold || showEdge.harvilleMid > leanThreshold)
    → 'lean'                              // leanThreshold default 0.05
else if (mlDrift > driftThreshold)
    → 'drift'                              // driftThreshold default 0.5
else
    → 'none'
```

Three important properties:

1. **Harville-only.** The classifier does not consider Henery edges,
   heuristic edges, or pWin in isolation. Henery is *displayed* in the
   UI as a side-by-side reference but does not influence the tag. (See §G
   for why this means "Henery SLAM_DUNK" in your spreadsheet is a
   labeling artifact, not a separate classifier output — see below.)
2. **It's a floor check, not a magnitude check.** `harvilleFloor > 0`
   means: under the worst-case companion scenario (the largest-pool
   companion drains most leftover), the projected payout still exceeds
   Harville's fair price. That's a *robust* edge by construction — no
   threshold parameter is involved.
3. **It's NOT a function of drift, pool depth, or edge magnitude.**
   Inputs are combined as a logical OR over Place and Show floors. There
   is no multiplicative scoring.

> ⚠ Your spreadsheet shows "65/102 race-model rows" tagged SLAM_DUNK,
> suggesting one tag per (race × model) pair. The classifier produces
> ONE tag per horse-row in the analysis, using Harville. If your export
> is showing two model-specific SLAM_DUNK tags, that's the export logic
> labeling rows based on whichever model produced the highest edge per
> row, not the classifier firing twice. Worth confirming in
> `lib/export/race-xlsx.ts` and `lib/export/race-day-xlsx.ts`.

### B.2 — How often does SLAM_DUNK fire historically? Plot ROI vs threshold.

**Not currently produced.** The classifier has no tunable threshold
parameter for slam_dunk — it's a strict `floor > 0` boolean. To answer
your "what happens if I raise it by 10/25/50%" question we'd need to
generalize the classifier:

```ts
floor >= floorMargin × fair        // floorMargin = 1.00 today (implicit)
```

Then sweep `floorMargin ∈ {1.00, 1.05, 1.10, 1.25, 1.50}` over the
settled history and plot realized ROI of SLAM_DUNK Place/Show tickets
at each setting.

**Hypothesis worth testing:** if 64% of races today are tagging
SLAM_DUNK, it likely IS too permissive — pari-mutuel pools at 60-min-out
or 10-min-out are noisy, and the projected payout for a thinly-bet horse
swings $5+ between refreshes. Some of those "floor > fair" tags are
genuinely measurement noise that disappears at close. Two follow-on
diagnostics:

- ROI of SLAM_DUNK at T-1:00 vs the *same horse's* SLAM_DUNK status at
  gates-open. If the tag isn't stable across the last 60 seconds, the
  signal is noise. (Requires the CLV snapshot from A.3.)
- ROI of SLAM_DUNK at T-1:00 conditional on pool depth. If small-pool
  tracks (MTH, lower-tier weekday cards) produce most false positives,
  add a hard pool floor to the classifier.

### B.3 — Are inputs linear or multiplicative?

Strictly logical OR. No linear or multiplicative scoring of pWin, drift,
or pool depth into the tag.

There's an argument the tag should be a small product score:

```
slam_score = max(placeFloorEdge, showFloorEdge)
           × min(1, poolDepth / poolDepthFloor)
           × stabilityScore   // share of last 3 refreshes with edge > 0
```

…and fire only when `slam_score > 0.05` or similar. But that's not the
current design. The current design assumes the floor check is
self-validating because it already accounts for the worst companion.
The empirical question is whether floor-edge alone, at our scrape
cadence, is well-calibrated. See §A.2.

## C. Exotic construction (where the bleed is)

This is the section with the largest gap between *what the diagnostic
question assumes* and *what the code actually does*.

### C.1 — How are exacta probabilities computed?

**They aren't, in our math layer.** Search `lib/math/` for any exacta
or trifecta probability function — there isn't one. Exotic tickets
are produced exclusively by `lib/ai/bet-planner.ts`, which sends the
race state (per-horse pWin, fair prices, edges, mid/floor, plus
pool sizes) to Claude Sonnet via a tool-calling schema and lets the
model pick tickets in prose. The model has no explicit formula; it
reasons from pWin, the WPS edge bundle, and pool sizes.

So when you ask "are we accounting for IIA?" the literal answer is:
"we're accounting for whatever the LLM, given pWin and edges,
chooses to do." That's not a probability model. It's a pattern-matched
allocation.

This is the most likely explanation for the catastrophic exacta hit
rates (4%, 8%) you cited. The expected exacta hit rate under a
naive Harville assumption for a typical 8-horse race with the AI
picking a small box on the top 2-3 horses by pWin should be in the
25-40% range. A 4% hit rate is consistent with the AI picking
exacta straight tickets with a wrong order, OR boxing horses too
deep into the longshot tail, OR both.

### C.2 — What's the EV of every exacta ticket we placed, per our own model?

**Not currently computed.** We don't have a Harville-conditional
exacta probability:

```
P(A wins, B 2nd) = p_A × p_B / (1 − p_A)        # Harville
                ≈ p_A × p_B^β / Σ_{k≠A} p_k^β   # Henery
```

…multiplied by an estimated exacta payoff from the exacta pool
divided by the bettor's share. Today's bet planner sees the total
exacta pool dollar amount, but NOT a structured estimate of payout
per (A, B) ordering — it has to infer pool slicing from prose
intuition.

To check "are we manufacturing −EV tickets" we'd need:

1. A Harville-conditional exacta probability matrix per race (n × n,
   excludes diagonal).
2. An exacta probable-payoff estimate per (A, B) — derivable from
   the live exacta pool if scraped (FDR exposes these as
   "probable payoffs"; if our adapter doesn't pull them today, that's
   a one-day add to the scraper).
3. Per-ticket EV: `P(A first, B second) × payoff − cost`.

Until that exists, the bet planner is flying blind on exotic EV.
This is the single highest-leverage fix for the exotic ROI problem.

### C.3 — Are we accounting for takeout and breakage in fair-value exotic calcs?

**For WPS**: yes — `placePayoutBand` and `showPayoutBand` apply
`net = total × (1 − takeout)`, plus breakage (floor to nearest $0.10)
and the $2.10 minimum-payout floor.

**For exotics**: we don't compute a fair-value exotic price at all, so
the question of whether takeout is subtracted is moot. The exotic
takeout rates ARE stored per-track in `lib/data/takeout-table.ts`
(exacta, trifecta, superfecta columns) — they just aren't consumed by
any math function yet. They'd plug straight into a Harville-conditional
exacta calculator the day we write one.

### C.4 — Cap on exotic tickets per race? Skip rule?

The bet planner has soft guardrails in its system prompt:

- Skip exactas when pool < $1,000.
- Skip trifectas when pool < $3,000.
- Default to WPS-only on starved-pool tracks.
- Total stake across all tickets ≤ $20.

No hard cap on ticket *count*. The 99-exacta-in-50-race count you cited
(~2/race per model) is the AI's allocation style under the current
prompt: when pool depth allows, it tends to spread small box bets. The
prompt nudges it toward concentration on stronger signals but doesn't
enforce a ticket cap.

Worth considering: a hard rule that exacta tickets are only allowed
when at least one Harville-conditional exacta EV is > 0 (post-takeout,
post-breakage). That cuts the AI's discretionary tickets to those with
mathematically defensible edge. Same shape as the WPS classifier — a
math gate, not a vibe gate.

## D. Stake sizing

### D.1 — Flat, Kelly, or other?

**Flat-budget, AI-allocated.** Hard rules:

- Total stake per race ≤ `$20` (`DEFAULT_STAKE_CAP` in `lib/ai/bet-planner.ts`).
- Each ticket ≥ `$2`, whole dollars only.
- Empty plan allowed when nothing has positive edge.
- The AI chooses the allocation across tickets within the cap.

There is no Kelly fraction parameter. There is no per-ticket sizing
formula. The "spend more on stronger signals" guidance lives in the
system prompt, not the code.

### D.2 — Implied Kelly vs actual?

**Not currently computed.** For a single-event bet with probability `p`
and decimal odds `o`, full Kelly is:

```
f* = (p × o − 1) / (o − 1) = (p × (o − 1) − (1 − p)) / (o − 1)
```

To compare actual stake to implied Kelly we'd need:

- A bankroll concept (we don't have one — the $20 cap is per-race, not
  fractional of bankroll).
- An odds-equivalent for each ticket (easy for WPS via fair payout;
  harder for boxed exotics where "odds" depend on which combo cashes).

The honest path here: keep the flat $20 cap but report, per ticket, the
fraction Kelly says we *should* have staked given our own probability
estimate (clamped to ≤ ½-Kelly for safety). If our actual stakes are
consistently above ½-Kelly on the longshot end and below on the chalk,
that's an allocation bias to fix.

### D.3 — Market impact of our $10 bet on closing odds?

**Negligible at our size.** A back-of-envelope check:

- Median Win pool on a mid-tier US track on a weekday: ~$50K.
- A $10 bet on one horse moves that horse's share of the pool by
  $10 / (current_pool_for_horse + 10).
- For a 3/1 favorite holding ~25% of the win pool ($12.5K), $10 is
  0.08% of that horse's pool — < 1bp shift in implied probability.
- Place/Show pools are smaller (~⅓ to ½ of Win), but the math is
  identical and the impact is still under 1%.

At larger tracks (NYRA, CD) the impact is even smaller. At thin
weeknight tracks (MTH, MNR, weekday FL races) on a horse with a
sparse Place pool, $10 *can* move that horse's projected Place
payout by 5%+ when the field pool is sub-$3K — but we already
skip those via the pool-depth guardrails in C.4.

Conclusion: market impact is not load-bearing for sizing at $20/race.
This is worth re-checking if we ever raise the stake cap.

## E. Per-track / per-condition calibration

### E.1 — Track / surface / distance / field-size / class breakouts

**Not currently produced.** We have:

- Per-track *takeout* rates (`TRACK_TAKEOUT` in `lib/data/takeout-table.ts`).
  These ARE used downstream — `analyzeRace()` resolves takeout per
  trackCode for the projected payout math.
- Per-track *priors*: none. The FLB α=1.06 is global. β=0.81, γ=0.65 are
  global. We do not condition any model on track, surface, distance,
  field size, or race class.

So when MTH went −66% in the data you cited, the diagnostic to run is:

1. Filter the settled history to MTH races.
2. Compute Brier/log-loss for Harville and Henery on MTH vs the rest.
3. If MTH is meaningfully worse, candidates for cause:
   - Smaller pools → noisier projected payouts → SLAM_DUNK floor check
     is firing on noise rather than signal.
   - Different bettor pool composition → FLB α might need to differ
     by track.
   - Track-specific takeout (MTH is 17%/19%/25% — already in the table,
     so already applied correctly).

### E.2 — Minimum sample size per track before we trust the projection?

**No such gate currently exists.** Today, an MTH R1 with a $400 Show
pool produces the same SLAM_DUNK calls as a CD R8 with a $200K Show
pool. That's wrong on its face — small-pool projections are dominated
by sampling noise of who happened to bet in the last minute.

Reasonable additions:

- Hard floor: skip projected payouts entirely when total pool < some
  cutoff (e.g. $5K for Place, $3K for Show). Currently we DO show
  numbers from any non-zero pool.
- Per-track confidence prior: if we have < N (say 50) historical
  races on a track, fall back to the global FLB α and don't apply
  any track-specific correction. Once N ≥ 50, blend toward the
  track's empirical α.

Both are post-historical-evaluator work — we need the evaluator
first to know what "track-specific α" would even be.

## F. Drift signal interpretation

### F.1 — How is drift computed and how does it enter the bet decision?

Computed in `computeMlDrift()`, `lib/math/ev.ts`:

```ts
mlDrift = (currentOdds − mlOdds) / mlOdds          // both decimal
```

Positive drift = horse lengthening (less money than expected).
Negative drift = horse shortening (more money than expected).

How it enters the betting decision:

- **In the classifier** (`classifySignal`): drift is the *tertiary fallback*.
  It only fires when SLAM_DUNK and LEAN both didn't fire. Threshold:
  `mlDrift > 0.5` (i.e. the horse is at LEAST 50% longer than morning
  line). It does NOT modify SLAM_DUNK or LEAN tags. It's a separate,
  lower-priority signal.
- **In the bet planner**: drift is included in the per-horse signal table
  passed to the AI as informational context (`drift -27%` etc), but the
  classifier-level signal that gets passed is the same single tag
  (`slam_dunk` / `lean` / `drift` / `none`).

> ⚠ **Important correction to one of your premises.** You wrote that
> "BEL R2 narrative cites '-27% drift' as justification for a SLAM_DUNK
> that lost." The classifier does NOT use drift to justify SLAM_DUNK.
> SLAM_DUNK fires strictly on Harville floor edge > 0 (§B.1). If the AI
> narrative used drift to *justify* a SLAM_DUNK tag, that's the LLM
> rationalizing after the fact — it has access to drift as a feature
> and is weaving it into prose. The drift number had no causal role in
> the tag. This is a *narrative quality* problem, not a signal-logic
> problem, but it confuses post-hoc analysis.

### F.2 — Plot ROI by drift bucket

**Not currently produced.** Same scaffolding as A.1. For each settled
race:

- Bucket each horse by drift in {< −50%, −50% to −25%, −25% to 0%, 0% to
  +25%, +25% to +50%, > +50%}.
- Compute realized ROI for Win, Place, Show, and exotic tickets
  separately, conditional on bucket.
- Same plot per model.

The conventional wisdom is that strong negative drift (shortening) is
the predictive direction — money is informed and trends toward the
horse with private info. The classifier currently triggers on the
OPPOSITE direction (positive drift / lengthening), under Ryan's
heuristic that lengthening creates value relative to the morning line.
Worth empirically testing which direction is actually +EV in our data.

Two things this plot would tell us:

- **Is drift sign predictive at all** in our scrape window, separate
  from the WPS pool signal?
- **Does drift *interact* with the WPS signal** — i.e. is "SLAM_DUNK
  with strong negative drift" better than "SLAM_DUNK alone"? If yes,
  we should add drift as a multiplier in a future scoring classifier.

## G. Architecture / structural

### G.1 — Are Harville and Henery the same pipeline differing only in finishing-order decomposition?

**Yes.** Both consume the same `probs` vector produced by
`selectProbsAndSource()` in `lib/math/index.ts`:

1. Source priority: win_pool → decimal_odds → uniform_fallback.
2. FLB correction (`p^α`, normalized) applied when source is win_pool or
   decimal_odds. Skipped for uniform.
3. The corrected vector feeds Harville AND Henery in parallel. The
   ONLY difference is the conditional-probability decomposition for
   2nd/3rd:
   - Harville: `p_j / (1 − p_j)` — renormalized uniform draw.
   - Henery: `p_j^β / Σ_{k≠i} p_k^β` with β=0.81, then `p^γ` with γ=0.65
     for 3rd.
4. Both produce a `pPlace[]` and `pShow[]` per horse, fair payouts
   computed from `2 / p`.
5. Same projected Place/Show payout bands (`placePayoutBand`,
   `showPayoutBand`) — which now use Harville companion weights for
   the *mid* by default (see `payouts.ts` — `harvillePlaceCompanionWeight`,
   `harvilleShowPairWeight`). This is worth noting: the "mid" is mildly
   coupled to Harville's conditional model regardless of whether you're
   reading Harville-mid or Henery-mid in the edge bundle. The companion
   *probabilities* are Harville's; the *fair price* uses the chosen
   model.

So model-vs-model deltas on PLACE and SHOW are interpretable: same
inputs, same projected payouts (mid), different fair prices via
different conditional-order math.

### G.2 — Recalibration cadence?

**Never.** All three calibration constants are hard-coded:

- FLB α: `1.06` (default; env-overridable via `FLB_ALPHA`).
- Henery β: `0.81` (`HENERY_DEFAULT_BETA`).
- Henery γ: `0.65` (`HENERY_DEFAULT_GAMMA`).

No training window. No fitting. They came from the math literature, not
from our scraped data.

Once the evaluator from §A exists, the natural extension is a script
that grid-searches (α, β, γ) over the settled history minimizing log-loss
per pool, with a holdout window for honest out-of-sample reporting. Until
then we're using literature priors as point estimates.

### G.3 — Can we run a blended model (Henery for place/show, Harville for win)?

**Trivially, yes — the data is already side-by-side.** `RaceAnalysis.rows`
contains `harville` and `henery` `ModelOutput` objects per horse, both
populated on every refresh. A blended strategy is a pure post-processing
choice:

```ts
const winFair    = fairPayoutPer2(probs[i]);          // raw pWin
const placeFair  = henery[i].placeFairPayout;          // Henery for place
const showFair   = henery[i].showFairPayout;           // Henery for show
```

…or any other combination. The bet planner already accepts a `ModelKey`
('harville' | 'henery') and the lock endpoint generates BOTH plans in
parallel. Adding a 'blended' model is mechanical:

1. Add `'blended'` to `ModelKey`.
2. In the bet-planner user message, write `blended.placeFair = henery.placeFair`,
   `blended.winFair = harville.winFair`, etc.
3. Run it alongside Harville and Henery, settle, compare ROI.

If your attribution analysis ($+96 Harville win + $+42 Henery place,
$+X for blended) holds up across more sample, the blend is the right
default. The infrastructure to test this exists today.

## Priority list — what to build first

Based on what's most likely to move the needle, in order:

1. **Historical evaluator script** (`scripts/evaluate-history.ts`) that
   reads `data/locked-recs.json`, joins lock-time analysis to settled
   outcomes, and emits a CSV with: race_id, track, horse, pWin,
   pPlace_h, pPlace_henery, pShow_h, pShow_henery, finishPosition,
   tickets, profit. Everything else in this priority list builds on
   this CSV. **One day.**
2. **CLV snapshot.** Modify `auto-settle` / `poller` to persist the last
   pre-race `RaceAnalysis` (within 30s of gates-open) alongside each
   locked plan. CLV reveals true edge ~10× faster than ROI. **Half a day.**
3. **Calibration plot + Brier/log-loss** (A.1, A.2) over the CSV. **Half a day.**
4. **Harville-conditional exacta EV calc** (C.1, C.2). Gate exacta
   tickets behind `exacta_EV > 0`. **One to two days** (depends on
   whether the FDR adapter already pulls probable payoffs).
5. **SLAM_DUNK threshold sweep** (B.2) — generalize the classifier to
   `floor >= floorMargin × fair`, sweep over history, pick the
   floorMargin that maximizes ROI on hold-out. **Half a day** once #1
   exists.
6. **Per-track diagnostics** (E.1, E.2). Pool-depth gate; per-track
   Brier/log-loss. **One day** once #1 exists.
7. **Drift interaction analysis** (F.2). Bucket-ROI plot; test
   drift × SLAM_DUNK interaction. **Half a day** once #1 exists.

## What you should NOT conclude from this doc

- That the models are broken. They might be — calibration could be off,
  thresholds could be too permissive — but we don't know yet, because we
  haven't measured. Today's 50-race ROI sample is too small to declare
  the model wrong on outcomes alone.
- That Henery is better than Harville (or vice versa). They differ in
  defensible ways, the literature mildly favors Henery for non-favorites,
  and we'd need the evaluator from §A to actually choose.
- That the AI bet planner is the problem. It might be — see §C — but the
  WPS classifier is mathematical, not LLM-driven, and the WPS results
  matter more for total P&L than the exotics in dollar terms.

The right next move is the evaluator script (priority #1). It unblocks
every other diagnostic question you raised.
